The system

How this platform is produced

The approach, the nine-stage pipeline, the eight agents, the data schemas, and the exact rules by which papers are scored and methods are categorized and counted. Blue blocks are verbatim quotes from the specification documents in docs/; every card names its source file.

One registry, many views docs/phd-architecture.md §1–§2

Facts live in exactly one place — seven JSON registry files. The Obsidian vault, the dashboard, this site and the transfer report are generated views of that registry.

"Nothing is written twice. Paper notes, method counts, pipeline diagrams, gap clusters: all generated from the registry. Edit the registry, regenerate the views." docs/phd-architecture.md, section 1

Claude Code · orchestrator

Runs all skills and agents; the only tool that writes the registry.

NotebookLM · second reader

All PDFs in one notebook; answers the five verification questions. Never writes anything.

Codex · second engineer

Builds and reviews webapp code against spec; two implementations surface bugs.

Antigravity · browser E2E

Clicks through every page in a real browser and files fix tasks.

Understand-Anything · graph

Generates the knowledge graph over the wikilinked vault.

Obsidian · human layer

Where the researcher reads, annotates and thinks; annotations promote back into the registry.

"Rule: Claude Code owns the registry. Codex and Antigravity own the webapp code. NotebookLM owns verification. No tool crosses lanes." docs/phd-architecture.md, section 2

The pipeline · S1–S9 docs/phd-pipeline.md §1–§2

Every paper is a tracked object that is always in exactly one state: candidate → triaged → acquired → extracted → verified → integrated, with failure branches rejected, unavailable and quarantined. Only verified and integrated papers ever render anywhere. Press a stage.

S1DiscoverClaude Code + NotebookLM›
Input
The literature-review brief: 2022 onwards, zero-shot focus, drones first.
Process
Search arXiv, Semantic Scholar, Google Scholar and venue proceedings; NotebookLM source discovery in parallel; snowball from reference lists of anchor papers.
Output
~100–120 rows in candidates.json with one-line relevance notes.
Failure path
Too few drone-specific hits in a category → widen to adjacent platforms and flag the thinness as a finding.
S2Triagefixed 0–100 rubric · ends at Gate 1›
"Recency: 2025 to 26 = 25, 2024 = 20, 2023 = 12, 2022 = 6 · Real-hardware evidence: strong = 25, moderate = 15, weak = 5, none = 0 · Zero-shot status: zero = 20, semi = 12, no = 0 · Venue tier and citation velocity: up to 15 · Drone relevance (or cross-platform anchor value): up to 15" docs/phd-pipeline.md, section 2, S2 (formatting condensed)
Output
Ranked list: top ~50 proposed, next 15 held as reserves.
Gate 1
"approve, veto, or swap papers. Vetoes recorded with reason." — the human decision no agent may take.
S3Acquirefilename = paper id everywhere›
Process
Download PDFs to docs/literature/pdfs/{paper-id}.pdf; upload the identical set to one NotebookLM notebook. The repo, NotebookLM, registry, vault and webapp all join on the paper id.
Failure path
Paywalled → arXiv version, author page, lab page. Still nothing → mark unavailable, promote the top reserve, log the swap. Credentials are never stored anywhere.
S4Extractbatches of 5, parallel, strict schema›
Process
One extractor agent per paper, five in parallel. Full schema entry per paper plus a pipeline stage array with provisional method tags.
Rule
"not stated" over guessing; every quantitative claim carries a page or section reference in a provenance field.
Failure path
Scanned or unreadable PDF → OCR pass; still failing → quarantine with reason.
S5Verify · the anti-hallucination gatefive fixed questions›

For each extracted paper, NotebookLM answers five fixed questions grounded only in that PDF — platform/hardware, zero-shot status, methods used, headline result, stated limitations — and the answers are compared field by field with the extraction. All five consistent → verified. Any mismatch → quarantined with the discrepancy logged.

"Why it works: two independent models reading the same PDF rarely hallucinate the same fact. Agreement is strong evidence; disagreement is a flag, cheap to check." docs/phd-pipeline.md, section 2, S5
S6Integratesingle writer merges the batch›
Process
The registrar merges provisional method tags through the alias table, recomputes every usage count and trend, and validates that every pipeline stage references a registered method id.
Failure path
Ambiguous method → decision logged in methods-decisions.md, applied consistently forever after.
S7Analyseruns at ≥80% integrated · ends at Gate 2›
Process
Methods split into essential (count ≥ 8), common (3–7), rare (1–2); per-method synthesis notes; limitations pooled and clustered with recurrence counts; hardware-feasibility and novelty filters applied.
Output
Ranked shortlist of 3 gaps + 1 recommended direction, each with a minimum-viable pilot sketch. A gap without a 3-week-feasible pilot is demoted.
Gate 2
The researcher picks the direction AND approves its pilot together.
S8Renderidempotent: run twice, same output›

Vault, knowledge graph and dashboard regenerate from the registry; Codex reviews generator changes; Antigravity runs the browser E2E checklist. Never hand-edit generated files.

S9Writeends at Gate 3›

Report sections draft only from the registry and synthesis notes; every claim carries a paper id.

"every claim resolves to a verified paper; 10 random claims per section spot-checked against NotebookLM. Fail → the sentence is rewritten or cut. No exceptions." docs/phd-pipeline.md, section 2, S9 / Gate 3 (arrow typography adapted)

How data is calculated & categorized docs/phd-pipeline.md §3

The usage counts are the headline analysis, so the registry must not double-count or fragment methods. Four rules keep the counting honest and reproducible.

Canonical ids + alias table

Every method gets one kebab-case id. "dynamics randomization", "DR over physical parameters" and "randomized dynamics" all map to domain-randomization-dynamics through the alias table — a new phrasing gets an alias entry, never a new method, unless it is genuinely distinct.

The split test

"two phrasings are the same method if swapping them in a paper's pipeline changes nothing about what was built. If the mechanism differs (latent adaptation vs online system ID), they stay separate."docs/phd-pipeline.md, section 3

Count = distinct papers

"Count = number of distinct papers, not mentions. A paper using DR in three places counts once for DR."docs/phd-pipeline.md, section 3

Every judgement logged

Each merge decision goes to methods-decisions.md with one line of reasoning, so the counting is reproducible and defensible in the viva. Example already logged: MonoRace's offline calibration refinement counted under system-identification.

The agents docs/phd-agents.md

The orchestrator plans, dispatches and checks gates; it never does bulk work. Agents write results to disk and return two-line summaries. Model economy: sonnet for bulk work, opus where judgement quality is the product. Press an agent.

"Single writer: extractors write only to staging; the registrar alone writes the registry. Parallelism never touches shared files."docs/phd-agents.md, section 4
01scoutsonnet · S1–S2 · writes candidates.json only›
"You discover and score papers. You never extract them. […] Never invent papers; every entry needs a resolvable arXiv id or DOI."docs/phd-agents.md, scout definition
02extractor ×5sonnet · S4 · writes staging only, 5 in parallel›
"You extract exactly ONE paper per invocation. […] 'not stated' instead of guessing. Never infer results. Every quantitative claim gets a provenance field (page or section)." docs/phd-agents.md, extractor definition
03verifiersonnet · S5 · sets verified or quarantined›
"Be strict: a near-match on numbers is a mismatch." docs/phd-agents.md, verifier definition

Also checks that the DOI or arXiv id resolves to the claimed title and authors.

04registrarsonnet · S6 · the ONLY registry writer›
"You own registry integrity. No other agent writes these files." docs/phd-agents.md, registrar definition

Maps aliases, applies the split test, recomputes counts and trends, validates referential integrity, and reports violations rather than silently fixing them.

05gap-analystopus · S7 · ranked gap shortlist›
"Be sceptical. Prefer a narrow, defensible gap over an impressive vague one. This feeds a transfer viva."docs/phd-agents.md, gap-analyst definition

Runs only at ≥ 80% corpus integration; a gap must recur in ≥ 3 papers and carry a 3-week-feasible pilot sketch.

06dashboard-engineersonnet · S8 · webapp code only›
"NEVER hardcode paper or method content in components; if content is wrong, the registry is wrong."docs/phd-agents.md, dashboard-engineer definition
07thesis-revieweropus · S9 · cold context, read-only›
"You are a sceptical transfer-viva examiner. You did not write this document and you owe it nothing."docs/phd-agents.md, thesis-reviewer definition

Sees only the finished text and the registry, never the drafting conversation. Outputs findings by severity; after each pass it generates the five viva questions the draft is least prepared for.

08ros2-px4-engineersonnet · I1–I3 · implementation track, from week 1›
"Never propose a real flight test without the staged ladder: unit test → simulation → SITL → HITL → tethered → free flight. Every rung gets a written pass criterion before the next."docs/phd-agents.md, ros2-px4-engineer definition (arrow typography adapted)

The registry schemas docs/phd-architecture.md §3–§5 · docs/phd-implementation.md §2

Seven JSON files, one writer each, referential integrity validated on every merge. The one join key: paper-id — the PDF filename, NotebookLM source, registry entry, vault note and site page all join on it. Press a file.

·papers.jsonwriter: registrar · the paper state machine›
Key fields
id · title/authors/year/venue · doi_or_arxiv (must resolve) · status (candidate…integrated) · platform[] · simulator_stack[] · methods[] → methods.json · zero_shot (zero|semi|no) · observation/action space · deployment_compute · results_summary · limitations_future_work[] (drives the gap analysis) · relevance_to_phd · pipeline_id → pipelines.json · provenance per claim
Rule
Only verified and integrated papers render anywhere; quarantined papers sit in a work queue, never in the analysis.
·methods.jsonwriter: registrar · the frequency analysis›
Key fields
id (canonical kebab-case) · aliases[] (merge table) · usage_count (distinct papers) · papers[] · platforms[] · trend (rising|stable|fading)
Rule
Single-use methods stay in the registry with count 1 — rare methods are where novelty may live.
·pipelines.jsonwriter: registrar · ground truth for counting›

Per paper, the ordered end-to-end pipeline (sim setup → training → transfer step → onboard runtime → hardware) as a stage array. Stage entries reference method ids, so pipelines are the ground truth for frequency counting.

·gaps.jsonwriter: registrar · where the thesis direction is decided›

Clustered limitations with recurrence counts and member papers, the hardware feasibility filter, the novelty check against counted methods, the ranked shortlist of 3, the selected direction and its pilot experiment id. A gap qualifies when it recurs in ≥ 3 papers.

·candidates.jsonwriter: scout · awaiting Gate 1›

Discovery output: scored candidates with rubric breakdowns, strength signals and citation counts (fetched at S2 via Semantic Scholar; low counts never penalize 2025-26 papers). Rejections are kept with reasons for audit. Currently seeded with 70 entries.

·hardware-map.jsonwriter: claude code · the feasibility filter›

Owned equipment (Holybro X500 V2 + Pixhawk 6X + Jetson Orin Nano, RTX A4000 workstation, DJI RoboMaster EP, Meta Quest 3, Bambu H2C) mapped to the methods and experiments each item enables. Every candidate method and gap is filtered against this file.

·experiments.jsonwriter: registrar · the experiment state machine›
States
planned → simulated → sitl → hitl → tethered → flown → analyzed → reported
Key fields
rq/hypothesis (falsifiable) · gap_id · methods_kept[] · novel_component · baseline · metrics with targets · seeds ≥ 3 · real trials ≥ 5 · pass criteria per rung · kill_condition stated in advance · results_ref · safety_review
Rule
No state skipped, no result without its rung's written pass criterion. Figures regenerate from committed logs.

The implementation track · I1–I3 docs/phd-implementation.md

Runs in parallel with the literature track from week 1, so the transfer report contains real hardware evidence rather than only a survey.

I1Infrastructure bring-upweeks 1–3 · 8 acceptance-tested items›

WSL2 + CUDA on the A4000, Isaac Lab, Aerial Gym, PX4 SITL + ROS 2 + XRCE-DDS, Jetson flash + TensorRT, Jetson–Pixhawk bridge, X500 bench bring-up (no props), logging conventions. Each item is done only when its acceptance test passes and the evidence is committed to experiments/logs/i1/.

I2Baseline reproductionweeks 3–6 · ends at a tethered flight›

Reproduce one published zero-shot pipeline end to end — the designated candidate is Learning to Fly in Seconds (open code, tiny compute): train 3 seeds, export ONNX → TensorRT with a numerical parity check, benchmark latency on the Orin Nano (the number every later method choice filters against), SITL, HITL, tethered flight. The tethered hover is the minimum Chapter 4 headline.

I3Pilot experimentweeks 7–9 · after Gate 2›

The minimum viable experiment on the selected gap: the I2 stack unchanged except the one novel component. One variable at a time; the comparison against the I2 baseline is the result. A negative pilot with a diagnosed cause is valid transfer evidence; a missing pilot is not.

Flight ladder, never skipped: "unit test → simulation → SITL → HITL → tethered → free flight. Every rung gets a written pass criterion before the next." docs/phd-agents.md · docs/phd-implementation.md §6 (arrow typography adapted)

Verification & the law docs/phd-pipeline.md §5

The standing rule of this project: everything published is quoted or verified; nothing is invented. Unverified items are labelled, never hidden.

Status vocabulary

web-verified independently checked with a verbatim quote on file · web-found the identifier resolves; claims not yet cross-checked · to confirm from model knowledge; must be existence-checked before use · S4-lite extracted from the arXiv page; full-PDF pass and NotebookLM verification (S5) still ahead. Reconstructed equations on paper pages are labelled "notation ours".

The provenance chain

"Report sentence → paper id → papers.json entry → provenance (page/section) → PDF in repo → NotebookLM verification record. If an examiner challenges any claim in the transfer report, the chain resolves in under a minute." docs/phd-pipeline.md, section 5 (arrow typography adapted)
